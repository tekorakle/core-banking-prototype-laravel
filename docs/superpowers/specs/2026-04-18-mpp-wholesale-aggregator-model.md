# MPP Wholesale / Aggregator Model — Design Spec

**Status:** Future / Unscheduled
**Date:** 2026-04-18
**Related:** `docs/superpowers/specs/2026-04-18-mpp-whitelabel-embed-model.md` (Option 3)
**Context:** VertexSMS partnership discussion — captured while context was fresh.

---

## Problem

The current MPP payment flow (Option 1, shipped) is account-less — a customer calls `/api/v1/sms/send`, gets a 402 with payment rails, pays via USDC/Stripe/Lightning, and retries with a payment proof. No Zelta account required.

This works perfectly for AI agents paying per-request. It doesn't work for **existing commercial customers of a Zelta partner** (e.g. VertexSMS's existing SMS customers) who:
- Already have a billing relationship with the partner
- Don't want to pay per-request at retail price
- Don't want a second vendor (Zelta) in their procurement pipeline

Today, those customers either (a) go direct to the partner and skip Zelta entirely — losing multi-rail payment benefits — or (b) become Zelta customers and double-register. Neither is good for the partner or for us.

## Goal

Let a Zelta partner consume Zelta APIs **wholesale** and resell to their own customers under their own billing relationship, without exposing Zelta to those customers.

## Design

### Partner tier

Add a `wholesale` partner tier (or extend Enterprise). Characteristics:

| Field | Value |
|-------|-------|
| Monthly fee | Negotiated (typically $0 base + per-unit) |
| Pricing model | Per-1000-units wholesale rate (e.g. $0.25 / 1K SMS) |
| Rate limit | Negotiated (default 10,000 req/min) |
| Billing | Monthly invoice, net-30 |
| Branding | None — partner is invisible to end users |
| SLA | Enterprise-grade |

Reuses existing `FinancialInstitutionPartner` infrastructure; just a new tier entry in `PartnerTier` enum + pricing config.

### API authentication

Partner uses a single server-side API key. No change from today's partner API. The partner's backend is the only caller — their customers never see Zelta URLs, Zelta API keys, or Zelta branding.

### Optional: sub-customer tracking

Partners may optionally pass `X-Subtenant-Id: <partner-internal-id>` on requests. Zelta:
- Stores the sub-tenant ID on the usage record (for audit / partner-side reconciliation)
- Does NOT create a Zelta user for the sub-tenant
- Does NOT apply per-sub-tenant rate limits or billing

This is opt-in and additive — partners who don't care can skip it entirely. Partners who want per-customer usage reports can query `/partner/v1/usage?group_by=subtenant`.

### Billing

Usage-based monthly invoice, generated by existing `partner:invoice:generate` command. Pricing:
- Wholesale tier: negotiated per-unit rate in `partner_pricing` table
- Existing tiers: fixed monthly + per-unit overage (unchanged)

Overage behavior: wholesale has no overage cap — partners pay-as-you-go with a high trust limit.

| Area | Effort |

| Add `wholesale` tier to `PartnerTier` enum + migration | S |

**Total estimate:** 1-2 weeks of engineering, assuming partner-tier infrastructure stays as-is.

## Tradeoffs

**Pros:**
- Partner keeps their customer relationship; no double-billing friction
- Zelta gets predictable B2B revenue from wholesale volume
- Leverages existing `FinancialInstitutionPartner` infrastructure — ~80% built already
- Partner's customers use partner's existing KYC/compliance; Zelta inherits partner's compliance posture

**Cons:**
- Zelta's brand is invisible — no direct customer funnel from partner traffic
- Partner can undercut Zelta's retail price, so wholesale rate must be lower than the partner's reselling price by enough margin
- Compliance: Zelta must trust partner's KYC on sub-customers (shift liability in contract)
- If a partner goes delinquent, all their sub-customers get cut off simultaneously

## Open Questions

1. Should sub-tenant tracking be mandatory or opt-in? (Recommendation: opt-in; mandatory creates friction)
2. Pricing floor: what's the minimum wholesale rate before the model stops making sense for Zelta?
3. Should we allow partner-branded error messages (e.g. partner's support email in 402 responses)? This overlaps with Option 3.
4. Compliance: does Zelta need to KYC the partner's sub-customers for Travel Rule? (Probably yes for payments >$1,000; defer to RegTech domain.)

## When to Build

**Trigger:** A second commercial partner asks for wholesale access. VertexSMS alone isn't enough signal — they're fine with Option 1. Wait for a repeat request pattern before committing engineering time.
